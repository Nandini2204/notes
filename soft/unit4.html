4.1 What is good Software Design?किसी भी software design करने से पहले उसे analyse किया जाता है specify किया जाता है उसकी software requirement को भी specify किया जाता है | software डिज़ाइन समय सर्वप्रथम तीन बातो को मुख्यतया शामिल किया जाता है टेक्निकली रूप से आवश्यक है:-
1.Design
2.Code generation
3.Test
इन्हे बनाने की तथा प्रमाणिकता करने की आवश्यकता भी होती है | सॉफ्टवेयर software design बनाने के लिए कुछ डिज़ाइन methods की आवश्यकता होती है जिनकी सहायता से डिज़ाइन task निम्न प्रकार की डिज़ाइन produce करती है 1.Data design
2.Architectural design
3.Component design

1.Data Design:-

इसके अंतर्गत डाटा structures के द्वारा analysis के दौरान domain model के द्वारा created information को transform किया जाता है | डाटा डिज़ाइन सॉफ्टवेयर architecture को design करने एक conjection की भूमिका निभाती है |

2.Architectural Design:-

यह सॉफ्टवेयर के मेजर structure element के मध्य relationship को बताता है | यह एक computer based सिस्टम का framework होता है | सिस्टम specification,analysis मॉडल आदि से लिया जाता है |

Interface design:-

यह हमें बताता है की सॉफ्टवेयर स्वयं के साथ किस प्रकार communicate करता है ? उस सिस्टम के साथ व्यव्हार केसा होगा जिसके द्वारा operate किया जा रहा है | एक interface डिज़ाइन flow of information को समाविष्ट करता है साथ ही व्यवहारगत प्रकार को भी specify करता है|
इसलिए data तथा control flow diagrams ,interface डिज़ाइन लिए आवश्यक informataion बहुलता को provide करता है |

Component – Level Design:-

यह aarchitectural software के structural elements को एक procedural description में transform करते है | इसके अंतर्गत information को control specification तथा state transition diagram से obtained किया जाता है|


Cohesion and Coupling:-अगर software का design अच्छा होता है तो software quality बेह्तर होती है तथा software design को analyze करने के बहुत सारे तरीके होते है जैसे-coupling, cohesion, factoring, system shape आदि.|

Cohesion:–

cohesion एक measurement (माप) है जो कि एक module के अंदर elements की intra-dependability की degree को डिफाइन करता है. अर्थात cohesion में यह देखा जाता है कि एक module के elements एक दूसरे से कितनी करीब से सम्बंधित है. अर्थात modules के elements की relationship एक दूसरे से कितनी गहरी है. जितना ज्यादा cohesion होगा उतना ही अच्छा software design होगा. Module का cohesion जितना अधिक होगा, उतना ही कम module के मध्य coupling होगी.


4.2.1 Classification of cohesiveness:-Cohesion निम्नलिखित प्रकार का होता है.
1- coincidental
2- logical
3- temporal
4- procedural
5- communicational
6- sequential
7- functional

1- coincidental cohesion:-

यह cohesion सबसे निम्न स्तर का cohesion होता है. यह तब घटित होता है जब module के elements के मध्य कोई relationship नही होती. यह तब होता है जब प्रोग्राम को छोटे modules में तोड़ दिया जाता है.

2- logical cohesion:-

एक मोड्यूल में logical cohesion तब होता है जब एक मोड्यूल के elements के मध्य लॉजिकल relationship होती है. तथा elements एक ही लॉजिकल class में रखे गए functions को परफॉर्म करते हैं.

3- temporal cohesion:-

यह cohesion जो है वह logical cohesion की तरह होता है परन्तु इसमें elements एक ही समय पर execute होने चाहिए. temporal cohesion लॉजिकल cohesion से उच्च स्तर का होता है क्योकि इसमें elements एक ही समय पर प्रोसेस होते हैं.|

4- procedural cohesion:-

जब मोड्यूल के elements एक साथ समूह में होते हैं तो वह एक task को पूरा करने के लिए एक sequence मै execute होते हैं इसी को procedural cohesion कहते हैं.

5- communicational cohesion:-

जब module के elements एक साथ group में होते हैं जो कि एक कार्य को पूरा करने के लिए sequence में execute होते हैं तथा ये सभी elements एक ही समय डेटा (इनफार्मेशन) पर कार्य करती है तब इसे communicational cohesion कहते हैं|

6- sequential cohesion:-

जब module के elements एक साथ एक group में होते हैं क्योकि एक element का output दूसरे element के लिए input की तरह होता है तो हमें sequential cohesion प्राप्त होता है.|

7- functional cohesion:-

यह cohesion का सबसे उच्चतम स्तर होता है module के सभी elements एक function को परफॉर्म करने से सम्बन्धित होते हैं.|

Coupling:-

coupling एक measurement (माप) है जो एक प्रोग्राम के modules के मध्य inter-dependability के स्तर को डिफाइन करता है. अर्थात यह एक प्रकार की measurement है जो कि यह डिफाइन करती है कि software के components एक दूसरे पर किस प्रकार निर्भर(depend) तथा interact करते है. coupling जितनी कम होगी software design उतना ही बेहतर होगा.|


4.2.2 Classification of Coupling:-Coupling निम्नलिखित प्रकार की होती है:
1- content कपलिंग
2- common कपलिंग
3- external कपलिंग
4- control कपलिंग
5- stamp कपलिंग
6- data कपलिंग

1- Content Coupling:-

content कपलिंग जो है वह सबसे उच्च स्तर का कपलिंग होता है यह तब होता है जब कोई module किसी दूसरे module की आतंरिक क्रियाविधि पर निर्भर होता है. इसका मतलब यह है कि दूसरे module में कोई बदलाव होगा तो उस dependent module में भी बदलाव होगा.

2- common coupling:-

इसे global कपलिंग भी कहते हैं यह तब घटित होता है जब समान global data दो modules के द्वारा share किया जाता है. इसमें दोनों modules में बदलाव आएगा जब global data में बदलाव किये जायेंगे तो.

3- external coupling:-

इस प्रकार कि कपलिंग तब घटित होती है जब external data format तथा communication protocol को दो modules के द्वारा share किया जाता है.|

4- control coupling:-

इस प्रकार की कपलिंग में, एक module दूसरे module के flow को नियंत्रित करता है तथा डेटा को एक से दूसरे module में भेजता है.|

5- stamp coupling:–

इस प्रकार की कपलिंग में बहुत सारें modules समान प्रकार के डेटा स्ट्रक्चर को share करते हैं तथा इसके अलग-अलग भागो में कार्य करते हैं.|

6- data coupling:-

दो modules के मध्य data coupling तब होगी जब डेटा केवल उनके ही मध्य pass होगा.|


4.3 Software design Approaches:-सॉफ्टवेयर analysis और डिजाइन के दो मुख्य Approaches हैं, अर्थात्, function-oriented approach और object-oriented approach। 
Here are two generic approaches for software designing:

Top Down Design:-

हम जानते हैं कि एक system एक से अधिक sub-system से बनी होती है और इसमें कई components होते हैं। इसके अलावा, इन sub-system और components में sub-system और components के सेट पर हो सकता है और system में hierarchical structure बनाता है। Top down design पूरे सॉफ्टवेयर सिस्टम को एक इकाई के रूप में लेता है और फिर कुछ विशेषताओं के आधार पर एक से अधिक sub-system या components को प्राप्त करने के लिए इसे विघटित करता है। प्रत्येक sub-system या components को तब एक system के रूप में माना जाता है और आगे विघटित किया जाता है। यह process तब तक चलती रहती है जब तक कि Top down hierarchical में सिस्टम का निम्नतम स्तर हासिल नहीं हो जाता। टॉप-डाउन डिज़ाइन सिस्टम के सामान्यीकृत मॉडल से शुरू होता है और इसके अधिक विशिष्ट भाग को परिभाषित करता रहता है। जब सभी components की रचना की जाती है तो पूरी प्रणाली अस्तित्व में आती है। जब सॉफ़्टवेयर solution को scratch से डिज़ाइन करने की आवश्यकता होती है तो Top down डिज़ाइन अधिक उपयुक्त होता है और विशिष्ट विवरण अज्ञात होते हैं।

Bottom-up Design:-

bottom up design model सबसे विशिष्ट और बुनियादी components के साथ शुरू होता है। यह बुनियादी या lower level के components का उपयोग करके higher level के components की रचना के साथ आगे बढ़ता है। यह तब तक higher level के component बनाता रहता है जब तक कि desired system एक एकल components के रूप में develop नहीं होती है। प्रत्येक higher level के साथ, अमूर्तता की मात्रा बढ़ जाती है। जब किसी सिस्टम को कुछ मौजूदा सिस्टम से बनाने की आवश्यकता होती है, तो bottom up strategy अधिक उपयुक्त होती है, जहां नए सिस्टम में मूल primitives का उपयोग किया जा सकता है। दोनों, टॉप-डाउन और बॉटम-अप approaches व्यक्तिगत रूप से व्यावहारिक नहीं हैं। इसके बजाय, दोनों का एक good combination उपयोग किया जाता है।


4.3.1 Function Oriented Design:-इसकी निम्न विशेषताए होती है :-
एक सिस्टम को इस प्रकार से देखा जा सकता है जैसे की वह किसी set of function को perform करता है | सिस्टम के इस high level को देखने मात्र से ही प्रत्येक function सफलतापूर्वक अन्य अधिक विस्तृत function में रेफिक्सेड कर दिया जाता है
Example:-
1. Suppose एक new library member को क्रिएट करना है जिसकी सहायता से लाइब्रेरी में एक व्यक्ति की membership को ऐड किया जाता है |
2. उस member को एक unique membership number assign किया जाता है | 
3. फलस्वरूप उस मेंबर की मेम्बरशिप बताने के लिए दिया गया charge एक बिल के रूप में print कर उसी मेंबर को सोपा जाता है|
4. उपरोक्त function को किये जाने के लिए निम्न चरण अपनाये जा सकते है या उनके भी मुख्य उप-कार्य निम्न है |
a) assign-membership-number
b) create member record
c) print Bill
इसी प्रकार से सिस्टम स्टेट को centerlized किया जाता है तथा उसे different function के बीच share किया जाता है|
Example:-
Create nrw member 
Delete member 
Update member record

Object oriented design:-

इस प्रकार के डिज़ाइन एप्रोच में सिस्टम को इस प्रकार से देखा जाता है की जैसे collection ऑफ़ object को | 
-यह सिस्टम state object के मध्य decentralized होता है | तथा प्रत्येक object स्वयं की state की information को mange करता है|
Example:-
एक library automation software प्रत्येकं library member अपना स्वयं का एक object रख सकता है जिसमें उसके स्वयं के डाटा हो सकते है तथा जिन वह कार्य भी कर सकता है | वास्तव में किसी object के लिए डिफाइंड function को किसी अन्य objects के डाटा के लिए change या refer भी नहीं भी नहीं कर सकते है | objects स्वयं के internal data को लिए हुए रखता है जिनकी सहायता से उनका state define होता है | 
– अन्य शब्दों में हम कह सकते है की each object कुछ classes के member होता है | यह class super क्लास से feature को inherit के रूप में भी प्राप्त कर सकती है | अर्थात अवधरणा यह है की object message passing के through एक दूसरे से communicate करते है|
